# DKTSS 评分指引 (DarkKnight Threat Scoring System)

**背景：**
传统 CVSS 评分过于学术，无法反映攻防实战中的真实威胁。本系统 (DKTSS) 通过量化**实战阻力** (Friction) 和**武器化程度** (Weaponization) 来校准基础分数，生成更贴合企业安全现状的风险评级。

在 `reporting/report_template.md` 环节，Agent 必须通过此规则计算漏洞的 DKTSS 评分，且禁止凭借主观感觉给出 `High/Low` 评级，必须在报告附带推导公式。

---

## 一、 DKTSS 核心计算公式

```text
DKTSS Score = Base(基础分) - Friction(实战阻力) + Weapon(武器化) + Ver(版本)
```

**最终评级映射：**
- **Critical (9-10分)**：必须立即修复。通常是外网无认证直接 RCE/脱库。
- **High (7-8分)**：必须在下个发版周期解决。
- **Medium (4-6分)**：常规安全风险。
- **Low (1-3分)**：防御纵深问题/理论漏洞。

---

## 二、 指标细化推演规则

### 1. Base (漏洞原生毁灭力基础分)

| 漏洞类型 | 成立条件判断 | Base 基础分 |
|---------|------------|------------|
| 命令执行 / 反序列化 | 可执行系统命令 | 10 |
| SQL 注入 | 此注入点允许读库、拖库或写文件、可获取管理员密码 | 8 |
| SSRF | 可以访问企业内网元数据或云环境 metadata API | 7 |
| SQL 注入 / XSS | 仅可由于逻辑限制读取部分无关数据、或可钓鱼劫持 Cookie | 6 |
| SSRF | 仅仅是普通的 HTTP 请求外发、无回显、被严格限制 | 4 |

### 2. Friction (实战阻力/门槛扣减)
*此项只接受负值或0。三个子项**叠加扣取**。*

1. **访问路径限制** (网络层)
   - 互联网暴露即可利用：`0` 分
   - 必须打入企业**内网**环境：`-2` 分
   - 需要**物理接触**机器：`-4` 分
2. **权限门槛限制** (认证层)
   - 无需任何认证（游客访问）：`0` 分
   - 需要一个普通用户身份登录后利用：`-1` 分
   - 必须拿到管理员/高权限角色账号才能利用：`-3` 分
3. **交互门槛** (利用步骤复杂度)
   - 全自动一键触发（零交互）：`0` 分
   - 必须等待管理员点击某个按钮/查看私信触发（弱交互）：`-1` 分
   - 必须通过社会工程学引诱用户下载特定配置并导入（强交互）：`-3` 分

### 3. Weapon (武器化成熟度补偿)
*漏洞被利用的难易程度。*

- 网上已有成熟稳定的 EXP (如 metasploit, nuclei 已内置)：`+1` 分
- 目前仅有手动复现的 PoC 或需要特定的冷门环境，且漏洞确认为 `CONFIRMED`：`0` 分
- 漏洞成立条件苛刻、由于代码限制或极强的并发要求极难复现（通常为 `HYPOTHESIS` 类漏洞）：`-2` 分

### 4. Ver (版本动态影响)
*默认填 0。供特殊情况调整，如影响的是即将废弃的老版本可标记 `-1`。*

## 三、 Agent 计算推演示例（报告必须包含的过程）

**示例情境**：在后台管理员控制台（需内网访问+高权限登录）处，发现一个 `Confirmed` 的通过拼接引发的 SQL 注入，可以使用 `sqlmap` 拖库。

**Agent 的标准输出推理（要求纳入报告）：**
- `Base = 8` （SQL 注入且可脱库）
- `Friction = -5` （必须在内网 `-2`，且需要管理员登录 `-3`，无交互限制 `-0`）
- `Weapon = +1` （可以直接使用 sqlmap）
- `Ver = 0`

> 计算：`8 - 5 + 1 + 0 = 4` 分。
> **结果判定：Medium (中危)。尽管是脱库级漏洞，但由于内网且高授权的重重阻力，利用几率大幅降低。** 

*(此推理链路解决了 AI 经常将高权限后台 SQL 注入夸大为最高红色告警的不实战通病。)*
