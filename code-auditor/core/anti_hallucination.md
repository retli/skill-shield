# 反幻觉 & 反确认偏差 — 详细规则

> 代码审计最大的风险不是漏报，而是误报。
> 误报会毁掉审计报告的可信度，浪费开发团队时间。

---

## Part 1: 反幻觉规则 (Anti-Hallucination)

### 核心原则

```
⚠️ 宁可漏报，也不误报
```

### 规则明细

#### R1: 文件存在性验证
```
✗ 不得猜测文件路径
✗ 不得假设"典型项目结构"存在
✓ 必须用 Glob/Read/Find 确认文件存在后才能引用
✓ 文件不存在 = 该文件不纳入审计范围
```

**违规示例**:
```
❌ "在 src/main/java/com/example/SecurityConfig.java 中..."（未读取过该文件）
✅ "使用 Glob 搜索 SecurityConfig，在 src/.../SecurityConfig.java 找到配置"
```

#### R2: 代码引用真实性
```
✗ 不得凭记忆编造代码片段
✗ 不得将不同文件的代码拼接为同一段
✓ 必须用 Read 工具返回的实际内容作为引用
✓ 引用必须标注准确的文件路径和行号
```

**违规示例**:
```
❌ "第 45 行: stmt.executeQuery(sql);"（实际第 45 行是别的代码）
✅ "UserDao.java:45: String result = stmt.executeQuery(query);"（Read 返回的实际代码）
```

#### R3: 技术栈一致性
```
✗ 不得报告与项目技术栈不匹配的漏洞
✗ 不得假设使用了未观察到的组件
✓ 必须基于 S2 项目画像中确认的技术栈
```

**违规示例**:
```
❌ "Spring Security 配置不当"（但项目使用的是 Shiro）
❌ "MyBatis ${} 注入"（但项目使用 JPA/Hibernate）
```

#### R4: 漏洞可达性
```
✗ 不得报告纯理论漏洞（Sink 存在但无可控 Source）
✗ 不得在无法完整追踪时标记为 "确认漏洞"
✓ 必须同时确认 Source 和 Sink 且追踪到完整路径
✓ 无法完整追踪时，标记为 "需进一步确认" 而非 "漏洞"
```

#### R5: 版本确认
```
✗ 不得在未确认版本号的情况下报告已知 CVE
✓ 必须从 pom.xml / requirements.txt / go.mod 等确认实际版本
✓ 版本在安全范围内 = 不报告
```

### 幻觉检查清单 (每个发现出报告前逐项核实)

```
□ 我是否用 Read 工具读取了包含此漏洞的源文件？
□ 我引用的代码是否与 Read 返回的完全一致？
□ 我引用的行号是否准确？
□ 项目实际使用了我提到的技术/框架/库吗？
□ Source 到 Sink 的路径是否有完整追踪证据？
□ 如果涉及已知 CVE，版本号是否已确认？
```

## Part 2: 反确认偏差规则 (Anti-Confirmation-Bias)

### 核心原则

```
⚠️ 方法论驱动，而非经验驱动
```

### 规则明细

#### B1: 禁止经验先入
```
✗ 不得说 "根据我的经验，Java 项目通常存在..."
✗ 不得因项目类型就假设一定有某类漏洞
✗ 不得因源码量大就跳过详细检查
✓ 每个 Sink 类型必须独立搜索和验证
```

#### B2: 禁止跳过低概率项
```
✗ 不得因 "不太可能" 就跳过检查项
✗ 不得选择性只检查 "容易出问题" 的地方
✓ 按 coverage_matrix.md 逐维度检查
✓ 发现 0 个问题也是有效结论
```

#### B3: 禁止过早收敛
```
✗ 不得在发现第一个漏洞后停止搜索同类漏洞
✗ 不得因已有足够多发现就减少后续维度的审计深度
✓ 同一 Sink 类型可能有多个不同路径/入口
✓ 必须在所有已规划维度审计完毕后才能进入报告阶段
```

#### B4: 等量严谨性
```
✗ 不得对某些漏洞类型的证据标准低于其他类型
✓ 所有漏洞必须达到相同的证据标准
✓ 每个"否定"结论（安全确认）也必须有证据支持
```

### 偏差检查关卡

```
在 Phase 5（深度审计）的每个维度完成后自问:
□ 我是否搜索了本维度所有已知的 Sink 模式？
□ 我是否对 Tier1 文件给予了最高注意力？
□ 我有没有因为发现了大量某类漏洞，而忽略了其他类型？
□ 我的每个"安全"判断是否也有证据支持？
```

## Part 3: 误报过滤规则

### 认定为误报的情况

| 情况 | 说明 |
|------|------|
| **已净化** | Sink 前有有效净化，且净化方式正确 |
| **常量参数** | Sink 参数为硬编码常量，非用户可控 |
| **内部调用** | Sink 仅被可信的内部服务调用 |
| **框架防护** | 框架已提供默认防护（如 Spring CSRF） |
| **测试代码** | 仅在测试文件中出现 |

### 降级处理

| 情况 | 处理 |
|------|------|
| 有净化但位置不理想 | 降为 Low |
| 需认证才可访问 | 严重等级降一级 |
| 有 WAF/限流保护 | 在报告中注明 |

### 严禁降级

| 情况 | 说明 |
|------|------|
| RCE 类漏洞 | 无论是否需认证，至少 High |
| 反序列化 | 有 Gadget 链可达 = Critical |
| JNDI 注入 | 即使内网 = High |
