# 安全控制建模方法论 — 轨道 A

> 控制建模法发现的是"缺失类"漏洞: 敏感操作 − 应有控制 = 漏洞

## 核心公式

```
缺失类漏洞 = 敏感操作 − 应有控制

1. 枚举所有敏感操作 → "这个系统能做什么敏感的事？"
2. 为每个操作推导应有控制 → "做这件事应该有哪些保障？"
3. 验证控制是否存在且有效 → "保障到位了吗？"
4. 缺失或不完整 → 报告漏洞
```

## 与 EALOC 的集成

**控制建模只在 Tier1 和 Tier2 执行**:

| EALOC 层级 | 控制建模策略 |
|-----------|-------------|
| **Tier1** ×1.0 | **逐端点审计**: 枚举每个 Controller/Handler 的敏感操作，逐一验证控制 |
| **Tier2** ×0.5 | **关键流程审计**: 只审计涉及数据修改、资金、权限变更的 Service 方法 |
| **Tier3** ×0.1 | **不做控制建模**（模型层不涉及控制逻辑）|

## Step 1: 敏感操作枚举

### 敏感操作分类

| 类型 | 说明 | 风险等级 |
|------|------|----------|
| **数据修改** | CREATE / UPDATE / DELETE 操作 | 高 |
| **数据访问** | 带 ID 参数的 GET（可能 IDOR） | 中 |
| **批量操作** | 导出 / 下载 / 批处理 | 高 |
| **资金操作** | 转账 / 支付 / 退款 / 充值 | 严重 |
| **权限变更** | 角色分配 / 权限修改 | 严重 |
| **外部请求** | HTTP 调用外部服务（SSRF） | 高 |
| **文件操作** | 上传 / 下载 / 删除文件 | 高 |
| **命令执行** | 调用系统命令 | 严重 |

### 通用快速识别方法

```bash
# 路由/端点枚举 (根据框架调整)
grep -rn "@PostMapping\|@PutMapping\|@DeleteMapping" --include="*.java"
grep -rn "@app.route.*POST\|@router.post\|def post" --include="*.py"
grep -rn "router.post\|app.post\|@Post(" --include="*.ts" --include="*.js"
grep -rn "Route::post\|Route::put\|Route::delete" --include="*.php"

# 敏感关键词
grep -rn "delete\|remove\|update\|create\|transfer\|pay\|refund\|upload\|download\|export\|grant\|assign\|approve" --include="*.java" --include="*.py" --include="*.go" --include="*.php" --include="*.js" --include="*.ts"
```

### 输出模板

```markdown
## 敏感操作清单

| # | 端点/方法 | HTTP方法 | 敏感类型 | 位置 | EALOC | 风险 |
|---|-----------|---------|---------|------|-------|------|
| 1 | /api/user/{id} | DELETE | 数据修改 | UserController:45 | Tier1 | 高 |
| 2 | /api/transfer | POST | 资金操作 | PaymentController:56 | Tier1 | 严重 |
| 3 | /api/export | GET | 批量操作 | ReportController:78 | Tier1 | 高 |
```

## Step 2: 安全控制推导

### 通用安全控制矩阵

| 操作类型 | 认证 | 授权 | 所有权 | 输入验证 | 并发控制 | 审计日志 |
|---------|------|------|--------|---------|---------|---------|
| 数据修改 | 必须 | 必须 | 必须(非管理员) | 必须 | 视情况 | 推荐 |
| 数据访问 | 视敏感性 | 视敏感性 | 必须 | 必须 | — | — |
| 批量操作 | 必须 | 必须 | — | 必须 | — | 必须 |
| 资金操作 | 必须 | 必须 | 必须 | 必须 | **必须** | **必须** |
| 权限变更 | 必须 | **超级管理员** | — | 必须 | — | **必须** |
| 文件操作 | 必须 | 必须 | — | **路径验证** | — | 推荐 |

### 控制矩阵模板

```yaml
敏感操作: DELETE /api/user/{id}
位置: UserController.java:45
类型: 数据修改
EALOC: Tier1

应有控制:
  认证控制:
    要求: 必须登录
    验证方法: 检查 @PreAuthorize / @login_required / Depends(auth)
    结果: ✅/❌

  授权控制:
    要求: 管理员或本人
    验证方法: 检查 hasRole / permission 注解
    结果: ✅/❌

  资源所有权:
    要求: 非管理员只能操作自己的资源
    验证方法: 检查 entity.ownerId == currentUser.id
    结果: ✅/❌

  输入验证:
    要求: id 必须为合法格式
    验证方法: 检查 @Valid / 类型约束
    结果: ✅/❌
```

## Step 3: 控制存在性验证

### 验证方法

```
对每个敏感操作:
1. 读取端点代码（Controller/Handler 方法）
2. 检查方法上的安全注解/装饰器
3. 追踪到 Service 层，检查业务逻辑中的权限验证
4. 检查数据库查询是否有 owner 过滤
5. 记录每个控制的存在状态
```

### 常见缺失模式 → 漏洞映射

| 缺失控制 | 漏洞类型 | CWE | 说明 |
|---------|---------|-----|------|
| 无认证检查 | 认证缺失 | CWE-306 | 匿名用户可访问敏感操作 |
| 无授权检查 | 授权缺失 | CWE-862 | 低权限用户可执行高权限操作 |
| 无所有权验证 | IDOR | CWE-639 | 用户可操作他人资源 |
| 无并发控制 | 竞态条件 | CWE-362 | 余额/库存可被竞态利用 |
| 无幂等性 | 重放攻击 | CWE-294 | 同一请求可重复执行 |
| 无 URL 白名单 | SSRF | CWE-918 | 服务端可请求内部服务 |
| 无路径边界检查 | 路径遍历 | CWE-22 | 可读写任意文件 |

## 反隧道视野 (Anti-Tunnel-Vision)

```
- 单一模块不得消耗 Step 3 超过 30% 的时间
- 同类文件 ≥3 个共享相同模式时，合并为 1 个发现
- 每完成一个模块，强制问: "还有哪些攻击面我没碰过？"
```
